# v1.6.0 真实合并门禁设计

## 背景与目标

v1.6.0 对应 `docs/ROADMAP.md` 的“真实合并门禁”阶段和 GitHub Issue #26、#27。本版本把 `quality_gate_status` 从单一 check-run 汇总升级为 PR 进入人工合并决策前的证据聚合器，同时把 `review_pr_against_standard` 升级为按任务类型评估 intent、scope、evidence、ownership、policy、fallback 和 security 的结构化 reviewer 辅助工具。

本版本不自动批准、评论或合并 PR，不修改 branch protection 或 ruleset，不绕过人工审批，也不引入新的高风险写工具。

## 设计原则

1. 保留现有工具名、输入和输出字段；新增字段均为向后兼容扩展。
2. GitHub 事实采集与门禁/审查规则分离，两个工具共享同一份 PR evidence。
3. 每个证据源独立降级；权限不足必须显式报告，不得猜测为通过。
4. 结论来自单一结构化结果模型，Markdown 与 `structuredContent` 不分别计算。
5. v1.6.0 只加入保守的内置规则；仓库级策略配置留给 v1.7。

## 模块边界

### `src/github/codeowners.ts`

从现有 `src/tools/review-pr.ts` 抽取：

- CODEOWNERS 解析
- gitignore 风格路径匹配
- conventional path discovery（`.github/CODEOWNERS`、`CODEOWNERS`、`docs/CODEOWNERS`）
- ownership routing gap 计算

该模块由 `quality_gate_status` 与 `review_pr_against_standard` 共同复用。现有最后匹配规则优先、`*`/`?`/`**` 的有界动态规划实现和 ReDoS 回归测试必须保留。

### `src/github/pull-request-evidence.ts`

统一读取并归一化以下只读证据：

- PR metadata：head SHA、base branch、draft、mergeable、labels、author
- check runs
- commit statuses
- reviews 与 requested reviewers
- classic branch protection
- applied repository rules
- CODEOWNERS 命中与审查情况
- GraphQL `closingIssuesReferences` 返回的关联 Issue

PR 基础信息是完整 PR 评估的必要前提。基础信息成功后，其余证据源独立读取并记录 `unverifiedSignals`；一个 API 的 403/404/5xx 不应抹掉已经取得的其他证据。

### `src/review/pull-request-review.ts`

承载纯逻辑：

- 文件分类
- `workType` 推断
- secret-like patch 扫描
- 七个审查维度的 findings
- test coverage signal
- release risk
- finding 排序与结论计算

该模块不直接读取 GitHub，便于用小型单元测试覆盖规则组合。

### 现有工具文件

- `src/tools/quality-gate-status.ts` 保留 schema、工具注册、兼容的 ref 模式、门禁决策和 Markdown 渲染。
- `src/tools/review-pr.ts` 保留 schema、工具注册、evidence 编排和 Markdown 渲染。
- `src/tools/workflow-permissions-audit.ts` 的 YAML/permissions 分析由 workflow/infra PR 审查复用。
- `src/tools/release-readiness.ts` 复用统一 CI 证据语义，修复零 checks 误判 passing 以及 pending CI 不阻止发布的旧行为。

## `quality_gate_status` 设计

### 输入兼容

- `pullNumber`：执行完整 PR 门禁评估。
- `ref`：保留现有调用方式，只评估 check runs 与 commit statuses。
- 新增可选 `blockingLabels: string[]`。默认值为 `blocked`、`do-not-merge`、`release-blocker`、`security-blocker`，大小写不敏感、精确匹配；调用方可传空数组关闭。v1.7 的仓库策略可在未来提供该值。

### Evidence 结构

`structuredContent` 保留 `contextLabel`、`headSha`、`categories`、`totalChecks`，并新增：

- `evidence.checks`：check-run 与 commit-status 的 passing/failing/pending/skipped 数量和明细
- `evidence.pullRequest`：draft、mergeable、base branch
- `evidence.reviews`：required、approved、changes requested、CODEOWNERS 满足情况
- `evidence.branchProtection`：classic/rulesets 是否可验证、实际 required contexts/reviews
- `evidence.labels`：全部标签与命中的 blocking labels
- `evidence.linkedIssues`：可验证的关联 Issue
- `blockers`
- `warnings`
- `nextActions`
- `degraded`
- `unverifiedSignals`

check runs 和 commit statuses 分别保留，不通过名称进行跨来源去重，避免掩盖两个独立 GitHub 信号。结论计算使用两者的合并事实。

### 结论优先级

当多种信号同时出现时，使用以下优先级：

1. `failing`
   - check run 或 commit status 失败/错误/超时
   - `mergeable === false`
   - 有效 review decision 为 `CHANGES_REQUESTED`
   - 命中 blocking label
2. `pending`
   - check/status 仍在运行
   - `mergeable === null`，GitHub 尚未完成后台计算
3. `needs_review`
   - PR 是 draft
   - required approving review 未满足
   - branch protection 要求 CODEOWNERS review 且未满足
4. `policy_gap`
   - 目标分支没有 classic protection 和 applied rules
   - 权限不足导致关键策略无法验证
5. `no_evidence`
   - 没有 check runs、没有 commit statuses，且没有其他足以支撑结论的可验证证据
6. `passing`
   - 存在有效证据，且所有已配置的强制策略均满足

缺关联 Issue、存在 CODEOWNERS 但未强制审查、仓库没有 required review 规则默认仅产生 warning。它们不是 PR 本身失败的充分条件。

## `review_pr_against_standard` 设计

### Work type

新增可选 `workType`，复用现有 `SdlcWorkType`：

- `docs`
- `feature`
- `bugfix`
- `refactor`
- `security`
- `release`
- `infra`

调用方传入时直接采用。未传入时按标签、路径、标题和正文进行保守推断，并返回：

- `workType`
- `workTypeConfidence`: `high | medium | low`
- `workTypeReasoning`

低置信度不阻止审查，但要提示调用方显式重试。

### Review finding

保留 `severity`、`category`、`description`、`suggestion`，新增：

- `dimension`: `intent | scope | evidence | ownership | policy | fallback | security`
- `paths: string[]`
- `reason`

一个 finding 可以覆盖多个路径，因此使用数组而不是单一 `path`。

### 任务类型规则

- docs-only：不要求代码单元测试，但 PR 正文必须说明 Markdown、链接、示例或命令验证方式。
- feature：要求测试文件，或在正文中给出明确且可审查的免测理由。
- bugfix：要求复现说明和非 snapshot-only 的回归测试证据。
- security：要求威胁、权限、密钥和安全验证说明。
- release：版本、发布配置、回滚说明和验证证据必须匹配。
- infra/workflow：检查触发条件、最小权限、失败路径与回滚说明；workflow 变更复用现有 YAML permissions 分析。

结构化输出新增：

- `releaseRisk`: `low | moderate | high | critical`
- `testCoverageSignal`: `adequate | missing | not_required | insufficient_evidence`
- `ownershipRoutingGaps`

现有 `conclusion` 继续使用 `pass | needs_changes | risky_but_acceptable`，避免 breaking change。

## Secret 与高风险路径规则

secret 扫描继续只检查 patch 中新增的赋值类模式，不因文档或注释出现 password/token/secret 关键词而误报。以下路径提高风险，但不必一律判失败：

- `.env` 与同类环境配置
- auth/token/credential 相关源码
- `.github/workflows/**`
- release/publish 配置
- lockfiles
- dist/build 产物

每个 finding 必须携带对应路径、风险原因和可执行建议。

## Degraded mode

每个外部信号记录为 verified 或 unverified。权限错误通过 `handleGitHubError` 安全化，不返回 token、响应头或原始敏感响应体。

- 非关键辅助信号失败：保留结论，增加 warning 和 `degraded=true`。
- branch protection/rules 无法验证：若没有更高优先级的明确失败/等待信号，结论为 `policy_gap`。
- linked issue 查询失败：不得报告“没有关联 Issue”，只能报告该信号未验证。
- ref 模式不查询 PR、review、ownership 或 branch protection，也不因这些信号缺失返回 `policy_gap`。

## Release readiness 相关修复

`release_readiness_check` 当前在零 check runs 时报告 passing，并允许 pending CI 时 `isReady=true`。v1.6.0 将：

- 同时读取 check runs 与 commit statuses
- 零 CI 证据返回 `unknown`/no-evidence 语义
- pending 或 unknown CI 令 `isReady=false`
- 保留现有 open bug 与 CHANGELOG 检查

该修复确保 PR 门禁和发布门禁不会对相同 GitHub 证据给出矛盾答案。

## 测试策略

所有行为变更遵循 RED-GREEN-REFACTOR。

### Quality gate

- 全绿 check runs
- status-only 的 success/failure/pending
- check/status 失败优先于其他信号
- 零 checks/status -> `no_evidence`
- required review 缺失 -> `needs_review`
- draft -> `needs_review`
- mergeable null -> `pending`
- mergeable false -> `failing`
- 分支完全未保护 -> `policy_gap`
- branch protection/rules 403 -> degraded `policy_gap`
- required CODEOWNERS 未满足与非 required routing warning
- blocking label 默认值、大小写和空数组关闭
- linked issue 存在、不存在和 GraphQL 失败
- Markdown 与 structured conclusion 一致
- ref 模式兼容

### Structured PR review

- 显式 `workType` 覆盖推断
- docs-only 有/无文档验证
- feature 有测试、无测试、有明确免测理由
- bugfix 有/无复现和回归测试
- workflow permissions 过宽
- `.env`、lockfile、dist、auth/release 风险路径
- 普通 secret 关键词不误报
- 真实 secret-like 新增赋值被发现
- findings severity 排序和 paths/reason/suggestion 完整性
- ownership routing gaps
- release risk 与 test coverage signal

### Regression and integration

- 原有公共字段继续存在
- smoke 注册所有工具成功
- release readiness 零证据、pending、status-only 场景
- runtime/package/lockfile 版本一致

测试不访问真实网络；继续通过显式 Octokit fake 注入 handler。
