# v1.6.0 Real Merge Gates Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver ROADMAP v1.6.0 by turning PR checks into an evidence-backed merge gate, adding work-type-aware structured PR review, aligning release readiness, and publishing complete documentation and version metadata.

**Architecture:** Move CODEOWNERS and PR evidence gathering into focused GitHub modules, then keep tool files as schema/orchestration/rendering boundaries. Pure evaluators compute gate conclusions and review findings from normalized evidence so Markdown and `structuredContent` cannot diverge. All existing tool names and fields remain available; new inputs and fields are additive.

**Tech Stack:** TypeScript 6 strict mode, `@modelcontextprotocol/sdk` v1, Zod 4, Octokit REST + GraphQL, Vitest 4, npm/Node.js 24.

- [ ] **Step 2: Write RED tests for structured findings**

Define:

```ts
export type ReviewDimension =
  | "intent"
  | "scope"
  | "evidence"
  | "ownership"
  | "policy"
  | "fallback"
  | "security";

export interface StructuredReviewFinding extends Finding {
  dimension: ReviewDimension;
  paths: string[];
  reason: string;
}
```

Cases:

- docs-only with verification -> no missing-code-test finding
- docs-only without verification -> evidence finding
- feature without tests and without a non-empty `No tests:` explanation -> high finding
- feature with a specific no-test reason -> no missing-test blocker
- bugfix without reproduction text -> high finding
- bugfix without non-snapshot regression test -> high finding
- workflow/auth/release/lockfile changes raise release risk
- secret keyword mention is clean
- added assignment-like secret produces a high security finding with path
- every finding has dimension, paths, reason, and suggestion

- [ ] **Step 3: Implement file classification, inference, and rule evaluation**

Export pure functions:

```ts
export function classifyPrFiles(files: PrFile[]): ClassifiedPrFiles;

export function scanPatchForSecrets(filename: string, patch?: string): StructuredReviewFinding[];
export function evaluatePullRequestReview(input: ReviewEvaluationInput): ReviewEvaluationResult;
```

Use explicit body patterns for reviewable evidence:

- docs verification: headings/phrases such as `Verification`, `Validated`, `markdownlint`, `link check`, or an executed command
- no-test reason: a `No tests:` or `Testing not required:` prefix followed by at least 10 non-whitespace characters
- bug reproduction: `Reproduction`, `Steps to reproduce`, `Before`, or equivalent explicit section
- fallback: `Rollback`, `Fallback`, or `Revert` with non-empty detail

Do not treat vague words like “tested” alone as adequate evidence.

- [ ] **Step 4: Compute release risk and test coverage signal**

```ts
testCoverageSignal: "adequate" | "missing" | "not_required" | "insufficient_evidence";
```

Critical secret/.env findings -> critical; high findings or workflow/auth/release changes -> high; medium-only -> moderate; otherwise low.
